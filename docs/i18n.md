# Internationalization (i18n)

Reverbic ships with two languages, Spanish (`Es`, the default) and English
(`En`). All user-visible strings are resolved at runtime through a small
translation layer in `src/i18n.rs`.

## Locale files

Translations live in two flat JSON files:

- `locales/es.json`
- `locales/en.json`

Each file is a single JSON object mapping a string key to its translated value:

```json
{
  "modal.tab.radio": "[ Radio ]",
  "hint.play": "Play",
  "config.setting.autoplay": "Autoplay last station"
}
```

Both files are embedded into the binary at compile time via `include_str!`,
parsed into `HashMap<String, String>`, and stored in `OnceLock` globals. If
either file is missing or contains invalid JSON, startup fails with an explicit
error — there is no partial-load mode.

## Key naming convention

Keys are dot-separated, namespaced by area, for example:

- `modal.tab.radio`, `modal.search.placeholder`, `modal.loading.spotify`
- `hint.play`, `hint.search`, `hint.close`
- `config.setting.<name>` — the label of a settings item
- `config.tooltip.<name>` — the help/tooltip text for that settings item

The **same key set must exist in both `es.json` and `en.json`**. When you add
or change a visible string, update both files together.

## Lookup and fallback behavior

The current language is held in an `AtomicU8` (`0 = Es`, `1 = En`) and can be
switched at runtime with `set_language`. Strings are resolved with `t(key)`,
whose fallback chain is:

1. Look up the key in the **current** language map.
2. If not found and the current language is not Spanish, look up the key in the
   **Spanish** map (Spanish acts as the base fallback locale).
3. If still not found, return the **raw key** itself.

Returning the raw key makes a missing translation visible rather than crashing,
but it is never an acceptable shipping state — add the real string to both
locales.

## Wiring settings labels, tooltips, and hints

Settings rows are driven by the `SettingItem` enum in `src/app/modal.rs`. Each
variant maps to:

- a label, produced by calling `t("config.setting.<name>")`
- a tooltip/help string, via a `tooltip_key()` that returns
  `"config.tooltip.<name>"` (resolved through `t` at render time)

Other UI labels and hints follow the same pattern: code references a stable key
constant and calls `t(key)`, so swapping languages re-resolves every string
without restarting. Because lookups happen on render, language changes apply
immediately.
